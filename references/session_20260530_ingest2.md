# Session 2026-05-30 (Second Pass) — Journal Ingest Findings

## Summary

Praxis journal ingest cron ran at 20:45 UTC on 2026-05-30. 66 unevaluated journals found and processed. 4 real behavioral signals extracted. 3 lessons extracted (1 high-confidence, 2 low-confidence). 0 new shifts activated (cap at 12/12).

## Key Issues Discovered

### 1. Sibling agent file contention

A sibling subagent (task `9f33d093`) was spawned by the system and attempted to write the **same** `/tmp/praxis_ingest.py` temp file simultaneously. The sibling's write overwrote our file mid-edit, introducing f-string escaping bugs (`}` characters eaten by the shell interpolation layer in the sibling's `write_file` call). This corrupted the script and would have caused a SyntaxError on execution.

**Root cause**: Both the parent Praxis cron job and a sibling subagent used `/tmp/praxis_ingest.py` as a shared temp path. No process-isolation for temp files.

**Fix applied**: Patch to SKILL.md — use unique temp filenames per run (`/tmp/praxis_ingest_{run_id}.py`) or write to `{agent_root}/tmp/` with unique names. We had to re-read the file after detecting the corruption and re-write it entirely.

**Also**: The sibling agent see the `_warning` field in write_file response: `"sibling subagent ... modified the file but this agent never read it."` This warning is critical — always re-read a file after a sibling warning before using it.

### 2. journals_evaluated.jsonl duplicates (121 found)

After the ingest, a dedup pass on `journals_evaluated.jsonl` found and removed 121 duplicate entries out of 5,112 total. These were legacy duplicates from prior ingest runs that had non-canonical IDs (missing `.json` extension, mixed formats).

**Impact**: Each duplicate represents a journal that was silently skipped on every subsequent scan. Entries with the canonical form AND legacy form both present meant the eval set had near-identical entries that didn't match either form perfectly.

**Fix**: Made pre-scan dedup MANDATORY (not just recommended) in `references/journal_ingestion.md`. The injection script was also deduping post-write, but doing it before the filesystem scan catches the scan-miss class of bug.

### 3. Observation event noise floor (62 of 66 journals = 94%)

The ingest script recorded an event for every journal processed, even when no failure signals were found. 62 of the 66 new events had `outcome_type: "observation"` and `context_summary: "No specific failure signals in ..."`. These are pure noise — they dilute pattern detection without adding signal.

**Root cause**: The script's logic created a default observation event for every journal that didn't match the failure-signal branch. This violates the "When NOT to record" rule in the SKILL.md.

**Fix**: Updated `references/journal_ingestion.md` with an enforcement rule: "If no signal is extracted, mark the journal as consumed in journals_evaluated.jsonl but do NOT create an event in events.jsonl." The observation event creation was removed from the production script logic.

### 4. Shift cap blocked by near-identical legacy shifts

The 12/12 active shift cap was full:
- 11 slots: near-identical generic shifts ("When {skill} encounters execution failures: verify tool availability and parameter validity") — per-skill variants auto-generated by a prior ingest run
- 1 slot: "After installing or restoring any cron script, verify executable permissions"

The high-confidence custodian execution lesson (12 events, full causal grounding) could not be activated because the cap was full of low-value duplicates.

**Fix**: Added mandatory shift activation dedup/merge step before cap check in SKILL.md. The rule: merge overlapping domain+phase shifts, increment reinforcement_count, only then check if cap is exceeded. If still full, expire oldest/lowest-reinforced-count shift.

### 5. Shell f-string escaping in sibling writes

The sibling agent's `write_file` output showed f-strings like `f\"{parts[0]}/parts[1]}/{fname}\"` — the closing `}` was consumed by the shell interpolation layer. This is the same "F-string backslash escaping" pitfall from the gotchas, but manifested through a different path: the sibling's write_file implementation or the JSON serialization layer ate the `}` characters.

**Confirmed**: The SKILL.md already has a gotcha about f-strings in `terminal(command="python3 -c '...'")`. This extends it to also cover `write_file` when the content contains f-strings being written by a concurrent agent. The safe approach is to use `"/".join(...)` instead of f-strings in any script that will be written via `write_file`.
