# Praxis — Full Gotcha Catalog

## Shift Management

- **Active shift cap is hard** — The 12-shift cap is enforced on every activation. When at cap, new shifts must merge with existing ones, replace a weaker shift, or be rejected.
- **Shift activation dedup and merge before cap enforcement** — Before proposing a new shift, check for semantically overlapping active shifts. Merge-first, check cap-second.
- **Shift decay without reinforcement** — Active shifts not reinforced in 14+ days auto-expire. The noon `praxis:decay_check` cron runs this daily.
- **Shift decay check must handle missing `last_reinforced_at`** — Legacy shifts (created before v3.0.0) may lack `last_reinforced_at` and `reinforcement_count` fields. When checking decay, fall back to `created_at` if `last_reinforced_at` is absent. In one decay check, only 2 of 9 shifts were visible because 7 legacy shifts had no `last_reinforced_at`. Use: `lra = s.get('last_reinforced_at') or s.get('created_at')` and treat missing `reinforcement_count` as 0.
- **Shift semantic dedup before activation** — Compare new shift's `shift_text` against all active shifts. If an active shift already addresses the same intervention, reinforce it instead of creating a variant.
- **Shift proposal dedup must check ALL lesson reference fields** — When building the set of lesson IDs already covered by shifts, checking only `lesson_id` and `lesson_ref` is NOT sufficient. Active shifts use heterogeneous reference fields: some use `lesson_id`, some use `source_lesson` (string), some use `source_lesson_ids` (array of strings), and some use `lesson_ref`. If the dedup only checks `lesson_id`, it will miss all shifts that reference lessons via `source_lesson` or `source_lesson_ids`, causing the proposal logic to create hundreds of duplicate proposed shifts. In one ingest run, this produced 133 spurious proposals that had to be manually expired. **Fix**: Build the covered-lesson set by scanning ALL reference fields:
  ```python
  covered_lesson_ids = set()
  for s in all_shifts:
      if s.get('status') != 'active':
          continue
      for field in ['lesson_id', 'lesson_ref', 'source_lesson', 'source_lesson_ids']:
          val = s.get(field)
          if val:
              if isinstance(val, list):
                  covered_lesson_ids.update(val)
              elif isinstance(val, str):
                  covered_lesson_ids.add(val)
  ```
  Use this set to skip proposal for any lesson already referenced by an active shift, regardless of which field carries the reference.
- **Shift reinforcement must be domain/phase-scoped** — Only reinforce shifts whose domain/phase actually matches the new events. Don't over-reinforce unrelated shifts.
- **Reinforcement text-matching precision** — Require BOTH domain overlap AND semantic relation. Keyword-only false positives pollute reinforcement counts.

## Lesson Extraction

- **Lessons require causal grounding** — `[LESSON] What: <pattern>. Why: <cause>. When: <conditions>`. Lessons without "why" get `confidence: low`.
- **Failure-phase tagging is mandatory** — Events from journal ingestion must include `failure_phase`. Untagged events can't participate in phase-aligned extraction.
- **Skip unknown/null domain lessons** — Do not extract lessons where domain is `unknown`, `None`, `null`, or empty.
- **Pattern detection must not rely solely on `context_summary` keywords** — Legacy events (pre-v3.0) often have empty or null `context_summary` fields. When grouping events for pattern detection, use `outcome_type == 'failure'` + `domain` as the primary matching signal, with `context_summary` keyword matching as secondary enrichment. In one ingest run, 6 of 9 domain events had empty context_summary, causing a real failure pattern (N=2+) to be missed on first pass. Always filter by `e.get('outcome_type') == 'failure' and e.get('domain') == target_domain` first, then enrich with keyword scanning on non-empty summaries.
- **Re-read events.jsonl after writing new events before running lesson extraction** — After appending new events to events.jsonl, re-read the file before the lesson extraction pass. Events written in the same run must be visible to the pattern detector. In this session, the first extraction pass found 0 lessons because a newly-written event wasn't visible until a second pass re-read the file.
- **Cross-skill phase lessons are too generic** — Patterns spanning 20+ skills produce non-actionable shifts. Record as observations only.
- **Lesson inference quality** — Gate `infer_cause` on `(domain is real skill name) AND (failure_phase in {planning, execution, response})`. Return `None` otherwise.
- **Lesson id field variance** — Production uses `lesson_id`; schema specifies `id`. Use `les.get('lesson_id', les.get('id', ''))`.
- **Ad-hoc scan scripts must deduplicate lesson keys** — Load existing lessons first, build key set, check before extracting.
- **Lesson noise gate at creation time** — Lessons with `signal_type` in `{"", "unknown", "?", "no_op", "forge_activity", "routine", "no_signal"}` are routine system noise, not behavioral patterns. Filter at Pass 2 (before writing to `lessons.jsonl`), not at shift proposal time. Ingest runs that skip this gate produce 7-13 noise lessons per run that must be manually cleaned. See `ingest-script-pattern.md` for the canonical `NOISE_SIGNAL_TYPES` set.

- **Reject empty stub lessons at extraction time** — Lessons with empty `lesson_text` or missing `what`/`why`/`when` fields are not "low confidence" — they are incomplete extractions that should be rejected immediately. Do not write them to `lessons.jsonl` where they accumulate as noise. In one ingest run, 3 empty stubs from prior sessions had to be manually rejected. If Pass 1 produces a stub that Pass 2 cannot ground (no events with meaningful summaries), reject it on the spot.

## Event Recording

- **Do not record observation events for signal-free journals** — Only record events when there is a real signal (not routine no-op journals).
- **Cross-skill corroboration** — When two skills detect the same fingerprint within 30 min, create one corroboration event, not two.
- **Same-skill temporal dedup** — Check events.jsonl for same fingerprint within 6 hours before recording.
- **Batch event ID collision** — Use `time.monotonic_ns()` or monotonically incrementing counter, not `datetime.now()` in loops.
- **Event/lesson/shift ID format mixing** — Always use timestamp-based IDs and verify uniqueness regardless of format.
- **Domain normalization** — Use canonical skill names, not raw filesystem directory names.

## Journal Ingestion

- **`journals_evaluated.jsonl` mandatory pre-scan dedup** — Deduplicate by `journal_id` before EVERY filesystem scan.
- **Near-duplicate journal IDs evade exact-match dedup** — Run normalization pass if line count differs from unique-ID count.
- **Unknown signal_type filter** — Skip events with `signal_type` in `("unknown", "?", None, "")` before pattern grouping.
- **Journal directory scan pattern** — Use `find -path` covering today and previous day, diff against `journals_evaluated.jsonl`.
- **Non-standard journal paths** — Some skills write outside canonical pattern. Walk ALL subdirectories and filter at evaluate time.
- **Stale eval entries** — Filter unevaluated list through `os.path.exists()` before processing.
- **Signal extraction — validate structure first** — Check field existence before keyword matching. Treat summary-only hits as `confidence: low`.
- **Finch `signals` vs `sources` schema mismatch** — The ingest pattern checklist (items 9-10) historically referenced `sources.*` for finch journals. Actual finch scan journals (e.g., `scan-1800.json`) use `signals.cron` for cron data, `signals.email` for email, etc. The `sources` key may be completely empty while all data lives under `signals`. In the 2026-06-07 ingest, the `signals.cron.new_errors[]` array (containing a real `weave:sync-contacts` model deprecation error) was initially missed because the script only checked `sources.*`. The signal was only caught indirectly via `tasks_added[]` containing "error", and was classified as `failure_keyword` instead of the more accurate `cron_errors`. **Fix**: Always check `signals.*` in addition to `sources.*` for finch journals. Extract `signals.cron.new_errors[]` as `cron_errors` type signals. Do not rely on `sources.*` alone.

- **`dir()` in exception handlers produces `NameError`** — The pattern `return signals, data if 'data' in dir() else {}` inside an `except` block is broken: if the exception occurred before `data` was assigned (e.g., `json.load` failed), `data` doesn't exist, and `'data' in dir()` may return a false positive from imported names. The correct pattern is simply `return signals, {}` — if we're in the exception handler, we don't have valid data regardless. Never use `dir()` to check for variable existence in exception handlers.
- **Dict-format summaries with success status = noise filter required** — Many custodian/finch scan journals store `summary` as a dict containing failure counts/data for *other* systems (e.g., `"error_jobs": 7, "error_breakdown": {"transient_429": [...]}`). When `status` is `"ok"`/`"success"`/`"completed"` OR `type` is `"observation"`, the scan itself succeeded — do NOT extract failure_keyword signals from its data fields. Only keyword-match dict summaries when the top-level status/type indicates an actual failure. Prevents false-positive events from successful scan reports.

- **Finch scan summary dict `json.dumps()` false positives from embedded filenames/references** — The signal extraction checklist recommends `json.dumps(summary_dict).lower()` for keyword scanning of dict-format summaries. But finch scan summary dicts contain `drive.notable[]` arrays listing filenames (e.g., "Metformin_ER_Exception_Continuation_2026-06-10"), `email.notes` with subject lines, and other reference data. These embedded strings can contain failure keywords ("exception", "error", "fail") that have nothing to do with system health — the journal reports all systems normal with no escalations. The existing "Dict-format summaries with success status" gotcha does NOT cover this: finch scans often lack a top-level `status` or `type` field entirely, so the "success status" gate never fires, and `json.dumps()` keyword scanning produces false positives. **Fix:** For finch scan journals specifically (skill directory = `ocas-finch`), skip `json.dumps()` keyword scanning of the summary dict entirely. Rely only on the structured signal paths: `signals.*`, `findings[]` (treat each finding string as informational/observation unless it contains explicit escalation language like "DEPRECATED", "CRITICAL", "action required"), and `tasks_added[]`. In the 2026-06-14 ingest, finch scan-0100 produced a false-positive `failure_keyword` from "Metformin_ER_Exception_Continuation" in a Drive notable-files list — a filename, not a system failure.
- **`find -not -path` glob pitfall**: Shell `find` with `-not -path "*/\.*"` may return 0 results even when matching files exist, due to shell glob interpretation of the `\*` pattern. The `*` in `*/\.*` can be expanded by the shell before `find` sees it. Either single-quote the pattern or, preferably, use Python `pathlib`/`os.walk()` for filesystem scanning. The skill already recommends Python scripts for journal scanning — this gotcha explains **why** the shell approach is unreliable.
- **`escalation_flagged` array noise in custodian journals**: Custodian light scan journals store internal tracking state as `"escalation_flagged": ["issue_id_1", "issue_id_2", ...]` arrays. These are NOT new escalations — they are references to previously-known issues being tracked. When signal-extraction code keyword-scans the journal JSON and hits the word "escalation" inside this array key, it produces false-positive `escalation/planning` events. In one ingest run, all 5 `escalation/planning` events were this noise. **Fix**: When scanning custodian journals, skip `escalation_flagged` arrays — only treat top-level `escalation_needed: true` or `escalations` arrays with `type: "new_item"` as real escalation signals. Do not keyword-match against JSON key names.
- **"Correction keywords in summary" false positives**: Some journal summaries contain the word "correct" or "fixed" in contexts that are NOT behavioral corrections (e.g., "correctly handled", "fixed-width format", "correction to prior estimate"). When the only signal is a keyword match on a summary string and the `execution_result.status` is `ok`, do NOT record a correction event. Require either `status: "partial"/"error"` OR an explicit `fixes_applied > 0` OR `actions_taken[].outcome` indicating a fix was applied. In one ingest run, 6 "correction" events (dispatch + elephas) were all keyword artifacts from routine nominal journals.

- **Top-level `summary` string false positives from scan journals**: Custodian light/deep scan and finch scan journals store a plain string `summary` at the top level (not inside `decision`). These summaries describe routine scan results and routinely contain failure keywords in non-failure contexts — e.g., "All 13 error-state jobs are transient HTTP 429 rate limits", "Calendar OAuth still expired", "Cron errors stable at 15 (all 429s)", "System healthy", "All clean. No new actionable issues." Keyword-matching these produces false-positive `failure_keyword` and `auth_failure` events. The existing "Dict-format summaries" gotcha only covers dict-format summaries, not plain strings. **Fix:** Apply a semantic suppression check (see `ingest-script-pattern.md` §Top-level summary string noise filter) that checks for suppress phrases like "all clean", "system healthy", "no new actionable", "transient", "consecutive_failures=0" before emitting `failure_keyword` signals from top-level summary strings. Only suppress when the ONLY signals are `failure_keyword` — if there are also `escalation` or `execution_error` signals, don't suppress. In the 2026-06-07 ingest, this filter would have prevented 5 false-positive events.

- **Summary signal suppression gap when mixed with non-summary signals**: The `should_suppress_summary_signals` function only suppresses `failure_keyword`/`auth_failure` when ALL signals are summary-derived. When a journal produces BOTH a legitimate non-summary signal (e.g., `correction` from `fixes_applied`) AND a summary-derived `failure_keyword` from the same summary text, the suppression is skipped — leaving the false-positive `failure_keyword` through. **Example:** A custodian light-scan with `fixes_applied: [2 fixes]` and `summary: "All other error jobs remain transient (cf=None/0). System healthy."` produces both `correction` (legitimate) and `failure_keyword` (false positive from "error" in summary). `should_suppress_summary_signals` sees `correction` and returns `False`, leaving the noise. **Fix:** After the primary suppression check, add a secondary filter: if `failure_keyword` signals exist alongside non-summary signals, check the summary for transient/stable phrases ("transient", "stable", "remain", "all errors transient", "no new") and remove the `failure_keyword` signals. This is narrower than the primary suppression — it only removes summary-derived keywords describing stable error states, not keywords describing new failures. Discovered 2026-06-20: custodian light-scan produced a false-positive `failure_keyword` that survived primary suppression because `correction` was also present.

- **Spot observation `type` field is case-sensitive**: Spot journals use `"type": "Observation"` (capital O), NOT `"observation"` (lowercase). The noise filter check `data.get("type", "") == "observation"` will NOT match. **Fix:** Use case-insensitive comparison: `data.get("type", "").lower() == "observation"`. Without this, spot observation journals bypass the no-op handler and fall through to signal extraction, producing spurious `all_skipped_observation` and `repeated_skip` events. Discovered 2026-06-13: spot sweep journal `sweep_20260613_100000.json` had `type: "Observation"` and was not caught by the lowercase check.

- **Spot sweep `failure_keyword` false positives from routine no-ops**: Spot journals with type "sweep" (or no type field) routinely report all watches inactive/skipped/deactivated. Summaries contain "inactive", "skipped", "deactivated", "zero active watches" — matching failure keywords. The existing spot observation handler only catches `type: "Observation"` (capital O), missing sweep-type and typeless spot journals. **Fix:** In `extract_signals_from_dict`, after the observation handler, add a secondary check for spot journals: if summary contains phrases like "all watches inactive", "zero active watches", "all skipped", "all deactivated", return `no_signal`. In the 2026-06-14 ingest, 6 false-positive events from spot sweeps required manual cleanup. See `ingest-script-pattern.md` for the code pattern.

- **Dispatch `auth_failure` false positive from dict summaries**: ocas-dispatch journals store `summary` as a dict with `gmail_auth_status: "unknown"`. When keyword-matched, "auth" triggers `auth_failure` — but this is a routine scan result, not a real auth failure. **Fix:** Never extract `auth_failure` from dict-format summaries. Only use structured signal paths (`escalation_needed`, `execution_result.status`) for auth failure detection. In the 2026-06-14 ingest, 2 false-positive dispatch auth_failure events required manual cleanup.

- **`all_skipped_observation` should not produce events**: When all spot results are skipped/expired/unautomated, the signal extraction may emit `all_skipped_observation`. This is a routine no-op state — NOT a behavioral signal. **Fix:** Either (a) filter `all_skipped_observation` out of `MEANINGFUL_SIGNALS` (it won't produce lessons), or (b) better: treat it as `no_signal` in the spot observation handler and skip event recording entirely. The event should only be recorded if there is a real failure pattern, not routine skip state.

- **Post-write dedup by `source_journal` loses multi-signal events** — The ingest script deduplicates events by `source_journal`, keeping only the earliest `recorded_at`. But a single journal can contain multiple distinct signal types (e.g., `escalation` from one finding + `execution_error` from another). Dedup by `source_journal` alone collapses these into one event, losing signals. **Fix**: Dedup by `(source_journal, signal_type)` composite key instead of just `source_journal`. This preserves distinct signal types from the same journal while still preventing true duplicates. See `ingest-script-pattern.md` for the known-limitation note. The signal extraction checklist in `ingest-script-pattern.md` produces raw keyword matches. If these are written directly to events without applying the noise filters from `journal_ingestion.md`, false-positive events accumulate and require manual post-hoc review. In one ingest run, a first-pass scan of 2 custodian journals produced 2 false-positive events ("401" in a success summary, "error" matching stale error_job counts) that had to be manually rejected. **Fix**: Build the noise filters from `journal_ingestion.md` (especially rules 1-5: escalation_flagged arrays, correction keyword false positives, dict-format summaries with success status, empty/None summaries, context_summary) directly into the extraction function. Only produce events that pass all filters. Do not produce raw matches for post-hoc review — the filters are deterministic enough to apply programmatically in a single pass.

- **Schema-ambiguous journals evade noise filter**: Some journals (e.g., ocas-elephas) use non-standard schemas without a top-level `status` field. When `data.get("status", "")` returns `""`, the noise filter `if status in ("ok", "success", "complete", "completed") and not signals` does NOT match, and the journal falls through without an eval_update. This causes two problems: (1) the journal is not marked as evaluated and will be re-scanned next cycle, and (2) if eval_updates are written in a batch append, the journal may inherit a wrong `action_taken` from a previous journal. **Fix:** After the signal extraction loop, if `signals` is empty, always append a `no_signal` eval_update explicitly — do not rely on status-field matching as the sole no-signal gate. See `ingest-script-pattern.md` for the code pattern.

- **Finch dict-style `findings` schema**: Some finch scan journals (e.g., `scan-0000.json`) use a `findings` dict (not array) where each key is a source name and values are dicts containing `status`, `detail`, etc. — e.g., `"findings": {"email": {...}, "system": {"status": "OK", ...}}`. The signal extraction checklist covers `findings[]` as array and `signals.*`/`sources.*` for finch, but does NOT explicitly cover `findings.*` as a dict with nested source data. **Fix:** When scanning finch journals, also iterate `findings.*` dict values — check each source's `status` field for error/failure values and `detail` string for failure keywords. Without this, a finch scan reporting `"system": {"status": "ERROR", "detail": "..."}` would be missed entirely.

- **`journals_evaluated.jsonl` error entries block re-processing**: When a journal is marked with `action_taken: "error"` in `journals_evaluated.jsonl`, it will NOT be re-processed by subsequent ingest runs (since it's already in the evaluated set). This means a script bug that errors on N journals permanently loses those journals' signals. **Fix:** After fixing a script bug, manually remove error entries from `journals_evaluated.jsonl` before re-running. Alternatively, add a re-processing mode that re-evaluates entries with `action_taken == "error"`. Discovered 2026-06-13: first ingest run's unpacking bug marked 5 journals as errors; they had to be manually removed from the eval file and re-processed.

- **No-op handlers must use `continue`, not `break`, in per-journal loop**: When a journal is identified as a no-op inside the `for canonical, fpath in unevaluated:` loop (e.g., forge scan returning `"clean"`, spot observation with all-skipped/expired results), the handler must write the `no_signal` eval_update and then `continue` to the next journal — NOT `break`. Using `break` exits the entire loop, leaving all remaining journals unprocessed AND skipping lesson extraction, shift proposal, decision logging, and the Praxis journal write. In the 2026-06-14 ingest, a `break` on the third forge no-op journal (out of 7 unevaluated) caused 4 journals to be silently lost and required a second full run to recover. Lesson extraction and shift proposal also did not run in the first pass. **Fix:** Both the forge no-op handler and the spot observation no-op handler (and any future no-op handler) must use `continue` after appending the eval_update. The only legitimate `break` in the per-journal loop is for `FileNotFoundError` (file disappeared between scan and processing). When adding a new no-op handler, always ask: "Does this `break` or `continue`?" — if it's `break`, change it.

- **Ingest script function return value count mismatch**: When refactoring `extract_signals` and `extract_signals_from_dict`, ensure ALL call sites agree on the number of return values. In one ingest run, `extract_signals_from_dict` returned 3 values `(signals, summary, status)` but the list-format handler in `extract_signals` tried to unpack 4: `entry_signals, entry_summary, entry_status, _ = extract_signals_from_dict(entry)`. This caused `ValueError: not enough values to unpack (expected 4, got 3)` for every journal processed. **Fix:** Define the return signature explicitly in a comment at the top of each function. Use a consistent 3-tuple `(signals, summary, status)` for `extract_signals_from_dict` and a consistent 4-tuple `(signals, summary, status, raw_data)` for `extract_signals`. After any refactor, grep for all call sites to verify unpacking matches.

- **Spot observation all-skipped detection must check `results[]` array**: Spot journals with `type: "Observation"` (capital O, case-insensitive) store per-watch results in a `results[]` array. Each result has a `status` field like `"skipped_inactive"`, `"skipped_platform_unavailable"`, or `"deactivated"`. When ALL results have statuses starting with `"skipped"` or `"deactivated"`, the journal is a routine no-op — NOT a behavioral signal. **Fix:** After the case-insensitive `type: "observation"` check, iterate `results[]` and check if all entries have skip/deactivate statuses. If yes, return `no_signal` immediately. Do NOT rely solely on the summary string — some spot journals have no summary at all, only the results array. The file grows by ~8 entries per 30-min cycle (~384/day). Without compaction, it becomes a performance bottleneck — dedup alone keeps it at the same size. The `journal_ingestion.md` spec says compact at >5,000 entries (remove entries older than 30 days). In one ingest run, the file had grown to 5,832 entries with no compaction applied, meaning every subsequent scan was processing 5,832+ dedup comparisons unnecessarily. **Fix**: Add compaction as step 1b in the ingest pattern (after dedup, before scan). This is now in `ingest-script-pattern.md` section 1b.

- **Ingest date window too narrow — scan ALL dates, not just today/yesterday** — The ingest script computes `today` and `yesterday` from `datetime.now(timezone.utc)` and only scans those two date directories. But cron jobs (especially forge journal-scan) write files to future-dated directories. These files exist on disk but are invisible to the scan. In the 2026-06-15 ingest, 15 forge June 15 journals were missed by the main run and required a manual secondary pass. **Fix:** Walk ALL date directories on disk — remove the `date_dir in (today, yesterday)` filter from the filesystem scan. The `journals_evaluated.jsonl` dedup is the actual gatekeeper; the date filter is an unnecessary optimization that causes misses.

- **`ocas-lucid` SKIP_DIRS causes re-scan waste** — `ocas-lucid` is in `SKIP_DIRS` but its journals are never marked as evaluated. Every cycle rediscovers them and skips them. **Fix:** Remove `ocas-lucid` from SKIP_DIRS. Lucid journals are routine degraded-mode scans that produce `no_signal` — scanning them is harmless and keeps the eval file complete.

- **Forge `files_processed` field is a list, not an int** — Forge journal-scan journals store `files_processed` as a list of processed file paths (e.g., `["file1", "file2"]`), NOT as an integer count. Any ingest script that checks `data.get("files_processed", 0) > 0` will crash with `TypeError: '>' not supported between instances of 'list' and 'int'` because Python cannot compare a list to an integer. The exception handler catches this as a `parse_error` event, producing a false-positive behavioral signal from a routine no-op journal.**Fix:** Either check `isinstance(data.get("files_processed"), list) and len(data["files_processed"]) > 0`, or better: remove the `files_processed` check entirely. The forge no-op handler (which checks `result` field) already filters routine journals before the activity detection runs. The `files_processed` check is redundant and should be removed from `extract_signals` to prevent this class of self-inflicted error. Discovered 2026-06-16 afternoon cron ingest: forged a `parse_error` event from a `result: "no-op"` journal because the type mismatch triggered the exception handler.

- **Forge no-op filter must use `startswith`, not exact match** — Forge result strings routinely include trailing detail text after the no-op keyword (e.g., `\"clean — no unprocessed VariantProposal or VariantDecision files found\"`). The exact-match check `result.lower().strip() in FORGE_NO_OP_RESULTS` does NOT match these longer strings, causing false-positive `forge_error` events. In the 2026-06-16 18:26 ingest, 2 false-positive events and 1 false lesson were produced before manual cleanup. **Fix:** Replace the exact-match check with a `startswith` loop:
  ```python
  def is_forge_no_op(result_val):
      if not result_val:
          return False
      r = str(result_val).lower().strip()
      return any(r.startswith(prefix) for prefix in (\"no_op\", \"clean\", \"no-op\", \"no_unprocessed_files\", \"no unprocessed\"))
  ```
  Or split on the em-dash and check only the first token: `r.split('—')[0].strip().split()[0] in FORGE_NO_OP_RESULTS`.

- **Forge no-op `FORGE_NO_OP_RESULTS` must include `"no unprocessed"` (with spaces)** — The `FORGE_NO_OP_RESULTS` set typically includes `"no_unprocessed_files"` (with underscores), but forge journal result strings use natural language with spaces: `"No unprocessed VariantProposal or VariantDecision files found. All scan locations empty."` The `startswith` check against `"no_unprocessed_files"` does NOT match `"no unprocessed variant..."`. **Fix:** Always include `"no unprocessed"` (with spaces) in the prefix set: `FORGE_NO_OP_RESULTS = {"no_op", "clean", "no-op", "no_unprocessed_files", "no unprocessed"}`. Without this, forge no-op journals with the natural-language variant produce false-positive `forge_error` events. Discovered 2026-06-16 23:14 ingest: `r_20260616_journal-scan-1781649877.json` produced a false-positive `forge_error` event because `"no_unprocessed_files"` (underscores) didn't match `"no unprocessed variant..."` (spaces).

- **New forge schema uses `action.result` instead of top-level `result`** — Newer forge journal-scan journals (post-2026-06-16) use a different schema with `action.result` (e.g., `{"action": {"result": "no_new_files"}, "decision": {"decision_type": "scan_complete_no_action"}}`) instead of the legacy top-level `result` field. The forge no-op pre-filter only checks `data.get("result", "")` at the top level — it does NOT check `data.get("action", {}).get("result", "")`. This causes new-schema forge journals to bypass the no-op filter entirely and fall through to signal extraction as unclassified journals. **Fix:** In `is_forge_no_op()`, also check the nested path:
  ```python
  def is_forge_no_op(journal_data):
      # Check top-level result (legacy schema)
      result = journal_data.get("result", "")
      # Check nested action.result (new schema)
      action = journal_data.get("action", {})
      if isinstance(action, dict):
          action_result = action.get("result", "")
          if action_result and is_forge_no_op_result(action_result):
              return True
      # ... existing top-level checks ...
  ```
  Discovered 2026-06-16 23:14 ingest: `journal-scan-1781650357.json` with `action.result: "no_new_files"` was silently skipped (no events, no eval_update with signals).

- **Finch `new_tasks_added` is a list, not an int** — Finch scan journals store `new_tasks_added` as a list of task dicts (e.g., `[{"id": "...", "title": "...", "severity": "medium"}]`), NOT as an integer count. ## Lesson Extraction — Operational Patterns

- **Lesson dedup by `lesson_id`, not by content** — Each ingest cycle re-extracts lessons from ALL events on disk. To avoid writing duplicate lessons, always read existing `lessons.jsonl` first, build a set of existing `lesson_id` values, and skip any new lesson whose `lesson_id` already exists. Dedup is by ID, not by text similarity.

- **`causal_grounding` field may be string or dict** — Legacy lessons (from pre-v3.0 ingest runs) may store `causal_grounding` as a plain string like `"Recurring X in Y phase"` instead of the dict `{"what": ..., "why": ..., "when": ...}`. When building shift text from a lesson's causal grounding, always guard: `cg = lesson.get("causal_grounding", {}); if not isinstance(cg, dict): cg = {}; why = cg.get("why", "No causal grounding available")`. A `AttributeError: 'str' object has no attribute 'get'` on this field crashes the entire shift proposal pass.

- **Ingest script must NOT early-exit before lesson extraction** — When `unevaluated` is empty (no new journals), the ingest script must still run lesson extraction and shift proposal against the existing event backlog. In one cron run, the script exited early with "No new journals to process" and skipped lesson extraction entirely, even though the backlog had 91 events that hadn't been fully consolidated into lessons. The fix: separate the "no new journals" path from the "no new events" path. Always run lesson extraction after the scan phase, regardless of how many new journals were found.
- **`skills_affected` empty → empty domain → merge failures** — When building `skills_affected` from events, if an event lacks `source_journal` or the skill can't be extracted, the resulting list may be empty. An empty `skills_affected` produces an empty `domain` on the shift, which causes the merge-overlap check to either miss real overlaps or falsely match on empty strings. Always default to the source journal's skill directory name: `e.get("source_journal", "").split("/")[0] or "general"`.
- **Reinforcement tracking drives merge decisions** — When multiple proposed shifts overlap with the same active shift, the active shift's `reinforced_count` increments. This count is used during merge to keep the strongest shift. Always update `reinforced_count` and `last_reinforced` on the active shift when reinforcing.

## Cron and Execution Context

- **`execute_code` is blocked in cron context** — Use `terminal()` to run Python scripts. Do NOT use heredoc (`<< 'PYEOF'`) — shell metacharacters inside Python code (e.g., `&` in comparisons) cause the shell to interpret them and the command fails. The reliable pattern: **write the script to a `.py` file via `write_file()`, then execute via `terminal(command="python3 /path/to/script.py")**. This works in both interactive and cron contexts.
- **`write_file` OVERWRITES JSONL files — experienced 2026-06-19** — During debrief generation, calling `write_file()` on `debriefs.jsonl` silently truncated 3 existing entries to 0 bytes. The file had 3 old debrief records that were lost. Recovery required re-creating the entries from session context. **Prevention**: For JSONL append operations, use `terminal(command="python3 -c \"...\"")` with explicit `open(path, 'a')` in the Python code. NEVER use `write_file` on any `.jsonl` file. This is the same root cause as the 2026-06-15 and 2026-06-16 incidents where `journals_evaluated.jsonl` was truncated. **Recovery pattern**: If `write_file` destroys a JSONL file, rebuild from filesystem scan: walk all journal files on disk, create eval entries with `action_taken: "rebuilt_from_disk"` and file mtime as `evaluated_at`.
- **`auth_failure` noise filter gap** — The `should_suppress_failure_keyword` function (ingest-script-pattern.md) only suppresses `failure_keyword` type. `auth_failure` signals from summary keyword matching need the same suppression. Use `should_suppress_summary_signals` (covers both types) in all new scripts. Discovered 2026-06-07: finch scan journal produced false-positive `auth_failure` from summary containing "All cron jobs healthy... OAuth expired... No new errors."
- **Variable set in loop, used after loop** — `summary`, `status`, and any other variable assigned inside a `for entry in data_list` loop but referenced after the loop (e.g., in noise suppression) MUST be initialized before the loop. Empty data_list → unbound variable → `UnboundLocalError`. Initialize `summary = ""`, `status = ""`, etc. before the entry loop.
- **Shell-embedded Python fails in cron** — Write `.py` file first with `write_file`, then run via `terminal()`.
- **Path traversal bug** — Use absolute paths, never `../` relative traversal from inside data directories.
- **`praxis_review.py` has wrong hardcoded DATA_DIR** — The review script at `scripts/praxis_review.py` hardcodes `DATA_DIR = '/root/.hermes/commons/data/ocas-praxis'` which is the **default profile** path. The actual indigo profile data lives at `/root/.hermes/profiles/indigo/commons/data/ocas-praxis/`. When run from the indigo profile, the script silently targets the wrong directory (or a non-existent one), producing empty results without error. **Fix:** Replace the hardcoded path with profile-aware resolution: `DATA_DIR = os.environ.get('PRAXIS_DATA_DIR', '/root/.hermes/profiles/' + os.environ.get('HERMES_PROFILE', 'indigo') + '/commons/data/ocas-praxis')`. Always verify the path exists and contains `events.jsonl` before proceeding.
- **F-string backslash escaping in write_file** — Use `.format()` instead of f-strings, or write script to file first.
- **F-string quotes inside `write_file` content** — When writing Python scripts via `write_file`, f-strings containing dict literals (`f"...{dict}..."`) can have quote conflicts with the outer string delimiters. The pattern `f"sources.{src_name}.status={src_status}"` inside a triple-quoted `write_file` content string can silently drop a closing quote, producing `SyntaxError: unterminated string literal`. Use `.format()` instead of f-strings within script content, or carefully escape. Always run `python3 -c "compile(open('script.py').read(), 'script.py', 'exec')"` after writing to catch syntax errors before execution.
- **Zero new journals is healthy** — `len(unevaluated) == 0` means the 30-min cron is keeping up.

## Status Field Gotchas

- **`"completed_with_errors"` is NOT a success status** — Some skills (notably ocas-weave) report `status: "completed_with_errors"` when part of a sync succeeds and part fails (e.g., inbound OK but outbound scope_failure). This is NOT the same as `"completed"` or `"ok"`. The noise filter's success-status check (`status in ("ok", "success", "complete", "completed")`) does NOT match `"completed_with_errors"`, which is correct — but the ingest script must then proceed to record the signal rather than falling through to an implicit no_signal. In the 2026-06-07 ingest, a weave google-contacts sync affecting 582 contacts was incorrectly classified as no_signal because the script treated unrecognized statuses as non-failure. **Fix:** Explicitly add `"completed_with_errors"`, `"scope_failure"`, and `"auth_failure"` to the failure status set checked during signal extraction.

- **`evidence` field may be dict or list** — Check `isinstance(ev, dict)` before accessing fields.
- **Legacy events use `id` instead of `event_id`** — Use `evt.get('event_id', evt.get('id', ''))`. Additionally, pre-2026-06-15 events use `type` instead of `signal_type`, `summary`/`description` instead of `evidence`, and `source_skill` instead of `skill`. ## Lesson Extraction Workflow

- **Two-pass lesson extraction is mandatory** — The initial extraction pass groups events by signal_type+phase and produces lesson stubs. A second "upgrade" pass must then add full causal grounding (what/why/when) to each lesson. Without the upgrade pass, all lessons are `confidence: low` and cannot produce shifts. Do not skip the upgrade pass. In one ingest run, the first pass produced 7 low-confidence lessons; the upgrade pass converted all 7 to high confidence with proper causal grounding.
- **Empty `skills_affected` produces empty domain** — When extracting lessons, if `skills_affected` is empty or missing, the resulting shift will have `domain: ""`. Empty domains cause the merge-overlap check to either miss real overlaps or falsely match on empty strings. Always default to the lesson's `skill` field or the source journal's skill directory name when `skills_affected` is empty.

The error message points to the line *after* the actual mismatch, making it hard to spot. **Fix:** After writing any `.py` file via `write_file`, always run `python3 -c "compile(open('script.py').read(), 'script.py', 'exec')"` before execution. Compute `active_count` from `all_shifts` at the top of the summary section, not inside any branch. This is the same class of bug as the existing "Variable initialization inside `if new_events:` block" gotcha — extend the pattern: ALL variables used in post-section summary/output must be initialized before any conditional.

- **Variable initialization inside `if new_events:` block** — When `active_count` (or any summary variable) is initialized inside an `if new_events:` block but referenced unconditionally after (e.g., `print(f"Active shifts: {active_count}/{MAX}")`), an empty `new_events` list causes `UnboundLocalError`. This is the same class of bug as `truly_new` and `remaining_proposals` — initialize BEFORE the conditional: `active_count = len([s for s in all_shifts if s.get('status') == 'active'])` at the top of the summary section, not inside any branch.

- **Variable initialization inside for/else is not unconditional** — A variable assigned only inside a `for` loop's `else` clause (or only inside the loop body) will not exist if the loop iterates 0 times. In one ingest run, `remaining_proposals` was only assigned inside `for proposal in new_proposals: ... else: remaining_proposals.append(proposal)`. When `new_proposals` was empty, the loop body never executed, and the variable was never defined. The subsequent `print(len(remaining_proposals))` raised `NameError`, crashing the script after all data writes had already completed (journal and decision log were lost). **Fix:** Initialize ALL variables that are referenced after a loop BEFORE the loop: `remaining_proposals = []`. This applies equally to variables initialized only inside `if` blocks — the existing `truly_new` gotcha is the same class of bug. Default to initializing every accumulator/reducer variable at the top of the function or section, not inside any conditional. — The `journals_evaluated.jsonl` compaction at >5,000 entries fires before the filesystem scan, but eval entries are appended after compaction. This means the effective threshold is 5,000 minus whatever the incoming batch size is. In one run, 5,999 entries passed the pre-scan compaction check (under threshold wasn't triggered since dedup brought it to exactly 5,999), then 7 entries were appended to reach 6,006 — without ever triggering compaction. **Fix:** Check compaction *after* appending eval entries, or use a lower pre-scan threshold (e.g., 4,500) to leave headroom for the incoming batch.

Calling `json.loads(open('file.jsonl').read())` raises `JSONDecodeError: Extra data` because the parser sees the second object after the first. **Fix:** Always read JSONL files line-by-line: `with open(f) as f: for line in f: obj = json.loads(line)`. Gotcha catalog shifts.jsonl, events.jsonl, lessons.jsonl, and journals_evaluated.jsonl are ALL JSONL format — never read any of them with a single `json.loads()` call. When the shift proposal loop iterates `all_lessons` and accesses `lesson['signal_type']`, it crashes with `KeyError`. The `get_lesson_id()` helper catches ID field variance, but the `signal_type` access in the proposal section does not. **Fix:** Before the shift proposal loop, filter to lessons that have `signal_type`: `proposal_candidates = [l for l in all_lessons if l.get('confidence') == 'high' and l.get('signal_type')]`. Always use `.get('signal_type')` (never direct bracket access) when reading from lessons whose schema lineage is unknown.

- **Lesson dedup matching on domain+phase alone is too broad** — When checking if an existing lesson covers a new pattern, matching on `domain + phase` without also checking `signal_type` will suppress distinct lessons that share a domain+phase but address different failure modes. In one ingest run, a new `ocas-spot execution_error` lesson was at risk of being suppressed by an existing `ocas-spot` lesson that covered the same phase but a different signal type. **Fix:** Always include `signal_type` in the dedup match: `e_domain == domain and e_phase == phase and sig_type in e_text_or_what`. Never match on domain+phase alone. When in doubt, extract the lesson — dedup belongs in shift activation (merge-before-cap), not in lesson suppression.

- **Proposed shift pool accumulation without activation** — Each ingest cycle can propose new shifts (up to the script's output) but if the active cap is immediately reached, those shifts accumulate as `status: "proposed"` indefinitely. In one run, the proposed pool grew to 159 entries, most of which will never activate because they're lower-priority than the 12 active shifts. **Fix:** Periodically (every 7 days), run a cleanup pass on proposed shifts: reject proposed shifts older than `proposed_ttl_days` (default: 7), deduplicate identical `shift_text`, and limit the total proposed pool to 50 entries.

- **Lesson content dedup by `(signal_type, failure_phase)` fails on case/phase variance** — When deduplicating new lessons against existing ones using `(signal_type, failure_phase)` as the key, the comparison is case-sensitive and exact-match. Existing lessons may use `failure_phase: "execution"` (lowercase) while new lessons use `"Execution"` (capitalized), or `"null"` vs `None` vs `""`. This causes the dedup to miss real duplicates, producing 21+ false-positive lessons from single-event patterns. In the 2026-06-16 ingest, this produced 21 bad lessons and 6 bad shifts that had to be manually reverted. **Fix:** Normalize both sides before comparison: `key = (signal_type.strip().lower(), str(failure_phase).strip().lower() if failure_phase else "")`. Build the existing-keys set with the same normalization. Also normalize at write time: store `failure_phase` as lowercase in new lessons.

- **`sorted(event_groups.items())` crashes on None signal_type** — When grouping events by `(signal_type, failure_phase)`, some legacy events have `signal_type: None` (from `get_signal_type()` returning `"unknown"` for events missing both `signal_type` and `outcome_type`, or from direct `None` values). Python's `sorted()` cannot compare `None` with `str`, raising `TypeError: '<' not supported between instances of 'str' and 'NoneType'`. **Fix:** Either filter out `None` signal_types before sorting, or use a sort key that handles None: `sorted(event_groups.items(), key=lambda x: (x[0][0] or "", x[0][1] or ""))`. Better: ensure `get_signal_type()` never returns `None` — it should return `"unknown"` as the fallback.

- **Shift append logic must write ONLY new shifts, not all shifts** — After proposing new shifts, the ingest script must append only the newly created shifts to `shifts.jsonl`, not rewrite or re-append all shifts (including pre-existing ones). In the 2026-06-16 ingest v6, the shift write loop iterated over `existing_shifts` (which included all previously loaded shifts) and wrote them all, duplicating 57 existing entries. **Fix:** Track which shifts are new (e.g., by keeping a list of new shift objects separately) and write only those: `for s in new_shifts_list: f.write(json.dumps(s) + "\\n")`. Do NOT iterate over the combined `existing_shifts` list.

- **Cap enforcement fails when active_shifts list is mutated during activation loop** — When the shift activation loop appends each newly activated shift to the same `active_shifts` list used for the cap check (`len(active_shifts) < CAP`), the list grows by 1 on every iteration, making the cap check always pass. In the 2026-06-16 ingest v9, this activated 5 shifts when the cap was already at 12. **Fix:** Use a separate variable for the cap count, not the mutable list: `active_count = len([s for s in all_shifts if s.get('status') == 'active'])` computed ONCE before the loop. Inside the loop, check `if active_count < CAP: activate; active_count += 1`. Do NOT append to the list being counted.

- **Lesson extraction must filter events with null/None/empty failure_phase before grouping** — When grouping events by `(signal_type, failure_phase)` for lesson extraction, events with `failure_phase` of `None`, `null`, `""`, or `"MISSING"` produce meaningless lessons like "Monitor and address failure during None phase". In the 2026-06-16 ingest v9, 86 events with `null` phase + 4 with `None` phase produced 26 low-quality lessons with invalid phases. **Fix:** Before the grouping loop, filter events: `valid_events = [e for e in all_events if e.get('failure_phase') and str(e.get('failure_phase')).lower() not in ('none', 'null', '', 'missing')]`. Use `valid_events` for lesson extraction. This filter should be applied AFTER `get_signal_type()` normalization but BEFORE grouping.

- **Forge `status: "complete"` is a routine no-op (new schema variant)** — Newer forge journal-scan journals (post-2026-06-16) use `status: "complete"` at the top level instead of `result: "no_op"` or `result: "clean"`. Both indicate routine healthy no-op (all scan directories empty, no files to process). The `is_forge_no_op()` pre-filter and the inline forge no-op handler only check the `result` field — they do NOT check `status`. This causes `status: "complete"` journals to pass through to signal extraction, where they may produce false-positive `forge_status_anomaly` events (since "complete" doesn't match the expected status values). **Fix:** In `is_forge_no_op()`, add `status` checking: `status = journal_data.get("status", ""); if status in ("complete", "completed", "no_new_files", "clean"): return True`. Also update the status-field noise filter to include `"complete"` in the success-status set: `if status in ("ok", "success", "complete", "completed") and not signals: return no_signal`. Discovered 2026-06-16: forge journal `r_20260616_journal-scan-1781648823.json` with `status: "complete"` produced a false-positive `forge_status_anomaly` event.

- **`success` signal_type from legacy events produces noise lessons** — Legacy events (pre-2026-06-15) use `outcome_type` instead of `signal_type`. When `get_signal_type()` falls back to `outcome_type`, events with `outcome_type: "success"` produce `signal_type: "success"`. These are routine healthy-state events, NOT behavioral patterns. The `NOISE_SIGNAL_TYPES` set in `ingest-script-pattern.md` must include `"success"` to prevent these from becoming lessons. In the 2026-06-17 ingest, 11 events with `signal_type: "success"` (from ocas-dispatch, ocas-custodian, ocas-elephas) produced a noise lesson and shift before cleanup. **Fix:** Add `"success"` to `NOISE_SIGNAL_TYPES` in `ingest-script-pattern.md`.

- **Inline ingest scripts must implement all documented gotcha filters** — When writing ad-hoc ingest scripts (rather than using the production `praxis_ingest_run.py`), it is NOT sufficient to follow only the basic scan/extract/append pattern. ALL gotcha-documented filters must be implemented: null-phase event filtering before lesson grouping, `NOISE_SIGNAL_TYPES` filtering before lesson write, forge no-op detection with `startswith`, spot no-op detection, summary string suppression, and `(source_journal, signal_type)` dedup. In the 2026-06-17 ingest, an inline script skipped the null-phase filter and the `success` noise type, producing 4 noise lessons and 2 noise shifts that required post-hoc cleanup. **Fix:** Before writing any ingest script, review the full gotcha catalog and implement every filter marked MANDATORY.

- **Journal directory name ≠ skill name in eval file** — The `journals_evaluated.jsonl` file stores journal IDs with the skill name prefix (e.g., `ocas-custodian/2026-06-13/deep-scan-...`), but the filesystem directory may lack the prefix (e.g., `custodian/` not `ocas-custodian/`). When constructing journal IDs from filesystem paths, a naive `skill_dir_name` produces IDs like `custodian/...` that never match the eval file's `ocas-custodian/...`, causing ALL journals to appear unevaluated (7,588+ false positives). **Fix:** Maintain a `DIR_TO_SKILL` mapping for known directories that lack the `ocas-` prefix, or normalize by checking if adding `ocas-` produces a match in the eval set. Always verify the mapping against actual eval file entries before running a full scan.

- **`data.get("action")` can be `None` (not dict)** — Forge journals may store `action: null` (JSON null → Python None). Calling `.get("result")` on `None` raises `AttributeError: 'NoneType' object has no attribute 'get'`. **Fix:** Check `isinstance(action_val, dict)` before accessing nested fields: `action_val = data.get("action"); action_result = action_val.get("result", "") if isinstance(action_val, dict) else ""`.

- **`data.get("action")` can be a string (not dict or None)** — Some forge journal-scan journals store `action` as a plain string summary (e.g., `"No unprocessed VariantProposal or VariantDecision files found. All locations clean."`) instead of a dict or null. Calling `.get("result")` on a string raises `AttributeError: 'str' object has no attribute 'get'`. This is distinct from the `None` variant above — the string variant contains the human-readable summary of what the scan did. **Fix:** Use `isinstance(action_val, dict)` guard before accessing nested fields: `action_val = data.get("action"); action_result = action_val.get("result", "") if isinstance(action_val, dict) else ""`. The string value of `action` can be treated as equivalent to the `summary` field for no-op detection purposes — if it contains phrases like "no unprocessed", "all locations clean", "no new files", treat as forge no-op. Discovered 2026-06-21: forge journal `r_20260618_journal-scan-1781824343.json` had `action` as a string, crashing the ingest script on first run.

- **`data.get("metrics")` can be `None` (not dict)** — Mentor and other journals may store `metrics: null`. Calling `.get("gap_detected")` on `None` raises `AttributeError`. **Fix:** Use `data.get("metrics") or {}` to coerce None to empty dict before accessing fields.

- **Batch limiting is mandatory for large unevaluated sets** — When thousands of journals are unevaluated (e.g., 7,588), processing all in one run exceeds timeout limits and consumes excessive context. The production `ingest-script-pattern.md` mentions batch limiting but doesn't enforce it. **Fix:** Always implement a `BATCH_LIMIT` (default 500) in ingest scripts. Sort unevaluated journals by mtime descending (newest first) and process only the first N. Track remaining count in the evidence record. The eval dedup ensures no journal is permanently missed — subsequent runs will pick up where the previous batch ended.

- **Eval file path mismatch causes all journals to appear unevaluated (2026-06-19)** — The `journals_evaluated.jsonl` file (21,082 entries) uses path format `skill/filename.json` (no date directory) while the filesystem scan produces `skill/YYYY-MM-DD/filename.json` (with date directory). The dedup cannot match, so every ingest reports 8,000+ journals as "unevaluated". **Root cause**: The eval file was rebuilt from a disk scan on 2026-06-15 using a different `path_to_journal_id` function. **Fix**: Normalize eval IDs at the start of each ingest run — for each entry, compute both possible path forms (with/without date directory, with/without `.json` extension) and check both. **Workaround**: Compare file mtime against `ingest_state.json:last_ingest_run` timestamp to detect new journals, bypassing the broken dedup.

- **Eval file format varies by profile — use dual-format matching** — The `journals_evaluated.jsonl` format differs across profiles and ingest runs. Some store IDs with `.json` extension, some without. The indigo profile as of 2026-06-18 stores IDs **without** `.json` (e.g., `ocas-mentor/2026-06-17/mentor-light-20260617T232020Z`). **Fix:** When building the evaluated set, store both variants: `evaluated.add(jid); evaluated.add(jid + ".json")`. Check both variants when testing membership. This is format-safe regardless of convention.

- **`decision` field can be a string, not just a dict** — Some dispatch (and other) journals store `decision` as a plain string summary instead of a dict with `execution_result`. Calling `decision.get("execution_result")` on a string raises `AttributeError: 'str' object has no attribute 'get'`. **Fix:** Always type-check: `decision = data.get("decision") or {}; if isinstance(decision, dict): exec_result = decision.get("execution_result") or {}; ...`. This applies to ANY field that can be either a dict or a string — always `isinstance` check before `.get()` chaining.

- **Legacy `outcome_type` events produce noise lessons after `get_signal_type()` fallback** — Legacy events (pre-2026-06-15) use `outcome_type` instead of `signal_type`. The `get_signal_type()` helper falls back to `outcome_type`, which can return values like `"success"`, `"routine"`, `"observation"` — all noise types. When these pass through lesson extraction, they produce noise lessons with valid-looking signal_types. The `NOISE_SIGNAL_TYPES` filter at lesson creation time catches most, but the fallback can also return `""` or `None` for events missing both fields, which then becomes `"unknown"` — also noise. **Fix:** After `get_signal_type()`, add an explicit noise check before lesson grouping: `sig = get_signal_type(evt); if sig in NOISE_SIGNAL_TYPES or not sig: continue`. This is in addition to the existing noise filter at lesson write time — defense in depth.

- **OCAS-priority batch limiting pattern** — When batch limiting, prioritize OCAS skill journals (which contain behavioral signals) over non-OCAS skill journals (which are mostly routine run logs). Sort unevaluated into two tiers: OCAS skills first (sorted by mtime descending), then non-OCAS (sorted by mtime descending). Process all OCAS journals + fill remaining batch slots with non-OCAS. This ensures behavioral signals are never left unprocessed due to batch limits. In the 2026-06-18 ingest, this approach processed 4,443 OCAS journals + 0 non-OCAS (batch was full with OCAS alone), leaving 218 non-OCAS for a follow-up run.

- **Post-ingest noise lesson/shift cleanup pattern** — When ingest runs produce noise lessons (from N/A signal_types, legacy schema variance, or incomplete noise filtering), a post-ingest cleanup pass should be run before the next ingest cycle. The cleanup: (1) read all lessons, remove any where `signal_type` is in `NOISE_SIGNAL_TYPES` or is empty/None, (2) read all shifts, remove any referencing removed lessons or having noise signal_types, (3) rewrite both files. This prevents noise accumulation across cycles. Track cleanup counts in the Praxis journal.

- **Cap saturation at 12/12 blocks all new shift activation** — As of 2026-06-18, the active shift cap was reached at 12/12 for the first time. When at cap, ALL new shift proposals are rejected regardless of lesson quality. The 14-day decay window is the only automatic release mechanism. Oldest shifts (June 14) auto-expire June 28. **Monitor cap saturation in debriefs** — when active shifts ≥ 10, flag approaching saturation. When at 12, list which signal types were rejected so Jared can manually expire low-value shifts if needed. The `stale_counters` and `tier2_open` signals from 2026-06-18 were captured as events but could not produce shifts due to cap.

- **Lesson re-extraction produces duplicate lessons against existing active shifts** — The lesson extraction Pass 2 re-processes the full event corpus each run, producing lessons for `(signal_type, failure_phase)` groups that already have active shifts. The merge-before-cap logic prevents duplicate shifts, but `lessons.jsonl` still grows with semantically duplicate entries. **Fix**: Before writing a new lesson, check if an active shift already covers the same `(signal_type, failure_phase)` key. If yes, skip lesson creation — the shift already encodes the lesson. In the 2026-06-18 ingest, 18 lessons were extracted but only 6 had new fingerprints; the other 12 were already covered by existing active shifts.

- **Custodian `stale_counters` and `tier2_open` are new signal types (2026-06-18)** — Custodian light scan journals produce `stale_counters` (when `cron_registry.stale_counters > 0`) and `tier2_open` (when `tier_2_open[]` has entries with `status: "open"`). `stale_counters` is low-severity (cosmetic, known pattern). `tier2_open` is medium-severity (latent bugs, known/recurring issues). Both are NOT new failures — they track known state. Add to `journal_ingestion.md` custodian signal type table.

- **`observation` signal type should be in NOISE_SIGNAL_TYPES** — Spot journals with `type: "observation"` produce `observation` signals. These are routine observation records, not behavioral patterns. Add `"observation"` to `NOISE_SIGNAL_TYPES`.

- **`ingest_state.json` is a JSON object, NOT JSONL (2026-06-20)** — The file is written with `json.dump(data, f, indent=2)` which produces multi-line JSON. Using `load_jsonl()` (which calls `json.loads()` per line) fails to parse it, causing `last_ingest_run` to default to epoch. This triggers a bulk re-scan of ALL unevaluated journals (4,500+). **Fix**: Read with `json.load(open(path))`, NOT line-by-line. If file doesn't exist, initialize with `{"last_ingest_run": null, "last_lesson_extraction_event_id": null}`. **Verification**: After reading, assert `last_ingest_run` is NOT epoch — if it is, use mtime-based detection only.
